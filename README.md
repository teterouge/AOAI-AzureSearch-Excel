# Leveraging OpenAI and Azure Cognitive Search for Excel Data

## Introduction

This repository demonstrates an early reference architecture for enabling conversational interaction with structured datasets (such as Excel and CSV files) using **Azure OpenAI** and **Azure Cognitive Search**.

The goal of the project was straightforward:

Allow users to ask natural language questions about structured datasets and receive answers generated by a large language model grounded in the data.

This work originated from experimentation within Microsoft's **Global Talent Intelligence** organization where our team relied heavily on Excel-based datasets for analysis and reporting. At the time, Azure OpenAI had just entered preview and there were very few working reference implementations showing how to integrate LLMs with structured enterprise data.

To address this gap, I built an internal prototype that combined:

- Azure OpenAI
- Azure Cognitive Search
- Excel / CSV ingestion pipelines
- Natural language querying over indexed data

The internal prototype proved useful for exploratory analysis and internal decision support. As Azure OpenAI adoption accelerated, Microsoft customer-facing teams began receiving similar questions from enterprise customers attempting to build “chat with your data” solutions.

A simplified version of this system was later **open sourced to provide customers with a starting point**, with proprietary components removed while preserving the core architectural pattern.

This repository represents **that simplified architectural snapshot**.

---

## Historical Context

This implementation was developed during the **first wave of enterprise GenAI adoption (2023)**.

At the time:

- Vector search capabilities in Azure Cognitive Search were still emerging
- Vector databases were not yet widely adopted
- Retrieval Augmented Generation (RAG) frameworks had not yet matured
- Many enterprise teams needed simple working architectures they could deploy quickly

Because of this, the system intentionally focuses on:

- structured data ingestion
- semantic search
- retrieval grounded responses using Azure OpenAI

Vector embedding support was considered but intentionally deferred to keep the implementation minimal and easier to understand.

As the ecosystem evolved, the internal system derived from this prototype was later revisited to **modernize the infrastructure and scale the architecture** to support larger datasets and broader operational usage.

This repository represents the **original architectural baseline**, not the later scaled version.

## Why Excel?

Excel documents are one of the most common data sources in many industries. It is one of my team’s most prolific sources.

The ability to chat directly with this data:

- streamlines decision making
- reduces time spent on manual data retrieval
- enables self-service access to routine reports
- allows analysts to interact with structured datasets using natural language

For our team this also helped **scale analytical outputs without scaling manual effort**.

## Key Features

1. **Data Engineering with Python**  
   Converts Excel sheets into structured JSON documents ready for indexing.

2. **Azure Cognitive Search Indexing with C#**  
   Creates a search index optimized for Excel-derived datasets.

3. **Integration with Azure OpenAI**  
   Uses Azure OpenAI alongside Azure Cognitive Search to enable natural language queries over structured data.

4. **Flask Interface for Testing**  
   A lightweight Flask backend with a simple UI for experimentation and validation.

## Excel Data Engineering Pipeline

To handle compatibility issues between Excel data and Azure Cognitive Search indexing requirements, the pipeline converts Excel files into structured JSON records using **Python, Pandas, and BytesIO**.

```python
# Iterate over all files in Container
blobs = container_client.list_blobs()
for blob in blobs:
    if not blob.name.endswith('xlsx'):
        continue

    blob_client = container_client.get_blob_client(blob.name)
    blob_data = blob_client.download_blob().readall()

    from io import BytesIO
    xls = pd.ExcelFile(BytesIO(blob_data))

    sheet_to_df_map = {}

    for sheet_name in xls.sheet_names:
        if sheet_name == "Overview":
            continue

        df = xls.parse(sheet_name)
        df['sheetName'] = sheet_name
        df['FileName'] = blob.name.split('/')[-1].split('.')[0]

        df.rename(columns=lambda x: re.sub(r'\W+', '', x.replace(' ', '_')), inplace=True)

        if 'Unnamed_0' in df.columns:
            df.rename(columns={'Unnamed_0' : 'Type'}, inplace=True)

        df['UniqueId'] = df['FileName'] + '_' + df['sheetName'] + '_' + df.index.astype(str)

        sheet_to_df_map[sheet_name] = df
```

Each row from each sheet becomes an individual JSON document indexed into Cognitive Search.

## Azure Cognitive Search Index

Index creation was implemented in **C#** due to difficulties configuring the preview Azure Search Python SDK for semantic search at the time.

```C#
SemanticSettings semanticSettings = new SemanticSettings();
semanticSettings.Configurations.Add(new SemanticConfiguration(
    "default",
    new PrioritizedFields()
    {
        TitleField = new SemanticField { FieldName = "FileName" },
        ContentFields = {
            new SemanticField { FieldName = "Field1" },
            new SemanticField { FieldName = "Field2" }
        }
    }
));
```

The indexer was configured to parse documents as **JSON arrays**.

Each Excel row becomes an indexed document, resulting in ~30,000 indexed records in the test dataset.

## Query Pipeline

Search queries use **semantic search** rather than vector retrieval.

```python
r = search_client.search(
    search,
    query_type=QueryType.SEMANTIC,
    query_language="en-us",
    semantic_configuration_name="default",
    top=3
)
```

Vector search was considered but intentionally deferred during early iterations to keep the system simple.

## Current Results

The system successfully enables natural language queries such as:

"Tell me about Titles..."

The response cites source records based on:

```
SheetName - FileName [source]
```

Because sheet names and file names are embedded into every indexed row, responses can be verified directly against the source Excel files.

## Known Limitations

The current architecture struggles to relate data located across **multiple sheets within the same Excel file**.

Resolving this likely requires:

- vector embeddings
- vector search
- improved prompt engineering

These improvements were planned for later iterations.

## Challenges and Solutions

The initial exploration began with the Azure Samples demo project, but the out-of-the-box architecture focused on PDF ingestion rather than structured datasets.

This led to a custom pipeline involving:

- Python data engineering
- Excel to JSON conversion
- Cognitive Search indexing
- Azure OpenAI retrieval integration
